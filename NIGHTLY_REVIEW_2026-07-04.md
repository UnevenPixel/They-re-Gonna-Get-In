## 3. Potential problems

Ordered roughly by how much it matters.

- **`scripts/Economy/Economy.gml`, `CanAfford` (~84‚Äì97) / `Purchase` (~112‚Äì119)** ‚Äî both loop over `struct_get_names(global.resources[_team])` and read/write the `Cost` struct via `struct_get`/`struct_set` by that same field name. If `global.resources` ever gains a resource key that `Cost` doesn't also declare, `struct_get(self, _res)` silently returns `undefined`: in `CanAfford`, `_resAmt < undefined` evaluates false, so the missing-key case is silently treated as "affordable"; in `Purchase`, it writes `undefined` back into `global.resources[_team][_res]`, corrupting that resource for the rest of the match. Currently benign only because `Cost`'s declared fields and `global.resources`'s initialized fields happen to match today (including the new `xp`/`fateTokens` fields) ‚Äî becomes a live bug the moment they diverge, with no defensive `?? 0` anywhere to catch it.
- **`scripts/UnitCombatHelpers/UnitCombatHelpers.gml` (load-bearing combat FSM), `UnitTryDealDamage` / `UnitTryFireProjectile`, and `scripts/StateMachine/StateMachine.gml`, `ChangeState` (~60‚Äì82)** ‚Äî `ChangeState` unconditionally clears `data = {}` on every state change, including a forced re-entry into the *same* state. `UnitTryDealDamage`/`UnitTryFireProjectile` both read `_machine.data.hitDealtThisSwing` with no existence check, relying entirely on `UnitBeginSwing` having run first. If any attack/siege/combat state ever force-re-enters itself (`ChangeState(currentName, true)`) without an immediate `UnitBeginSwing` call, this reads `undefined` (falsy) instead of the prior swing's `true`, silently permitting an extra hit that frame. Not confirmed as a live bug ‚Äî the specific state files that would trigger it weren't all in the same review batch ‚Äî but worth a direct check of whether any attack/siege/combat state does a same-state forced re-entry.
- **`scripts/UnitStateSiege/UnitStateSiege.gml`, `Siege_Step` (~lines 157, 202)** ‚Äî uses `_unit.attackAggroRadius ?? 96` as a defensive fallback, but `Combat_Step`, `Defend_Step`, `Guard_Step`, and `Attack_Step` all read `_unit.attackAggroRadius` directly with no fallback, and `UnitDefinitions.gml` confirms it's always set. Either the field can legitimately be missing sometimes (in which case the other four states have a latent bug), or it can't (in which case this is dead defensive code masking what should be a hard error) ‚Äî worth deciding one way and making it consistent.
- **`objects/oBuildingPlot/Create_0.gml`** ‚Äî see the empty-file note in the Syntax section above; `Step_0.gml` reads `blocked`/`inside` with nothing in this file (or an inherited parent, if one exists) to set them first.
- **`scripts/GatherScripts/GatherScripts.gml`, `_FindNearestEnemyInSweep` (~line 80) / `_FindNearestEnemy` (~line 148)** ‚Äî leading-underscore function names, which elsewhere in this codebase denotes a local variable, not a function. Both are already marked in their own comments as superseded/dead code now that `ChooseCombatTarget` exists, so low priority, but worth a naming cleanup if/when they're removed.
- **`scripts/OrderWiring/OrderWiring.gml` (load-bearing order/FSM wiring), `RegisterAllOrders`, ~lines 16‚Äì19, 50‚Äì54, 70‚Äì72** ‚Äî none of the `onIssue` callbacks guard with `instance_exists` before dereferencing `.defendTarget`/`.attackBuildingTarget`/`.fsm` on units in the passed-in list. Likely safe today since callers pass a freshly-filtered live list, but there's no defensive check in this file itself.
- **`scripts/ProjectileScripts/ProjectileScripts.gml`, `ProjectileInit`, ~line 63** ‚Äî `_proj.team = variable_instance_exists(_owner, "team") ? _owner.team : TEAM.PLAYER` silently defaults a teamless owner to `TEAM.PLAYER` with no log ‚Äî could mask a real bug where a projectile ends up misattributed to the player's side.
- **`scripts/UnitCombatHelpers/UnitCombatHelpers.gml`, `UnitEnterCombat`, ~lines 77‚Äì81** ‚Äî no `instance_exists`/`variable_instance_exists` guard on `_target` before assigning `_unit.combatTarget = _target`. Safe today because the only caller (`ApplyDamage`) already checks, but fragile if a second caller is added without the same guard.
- **`scripts/UnitDefinitions/UnitDefinitions.gml`, `UnitApplyDefinition`, ~line 184** ‚Äî `_unit.maxSpeed = _def.maxSpeed;` is a dead/duplicate write per the function's own comment ("nothing currently reads it"). Harmless, just legacy redundancy worth cleaning up.
- **`scripts/TrainingScripts/TrainingScripts.gml`, `TrainingSpawnUnit`, ~lines 183‚Äì194** ‚Äî no `instance_exists(_building)` guard before reading `_building.trainsUnit`/`.team`. Low risk (same-instance caller) but inconsistent with the guarded style used elsewhere.
- **`scripts/UnitSelection/UnitSelection.gml`, `EndDrag` (~134‚Äì166) and `UpdateTargeting` (~237‚Äì270)** ‚Äî both read `.team` right after a spatial query (`collision_rectangle_list`/`instance_position`) with no `instance_exists` guard in between.
